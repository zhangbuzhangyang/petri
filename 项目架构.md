作为 Project Petri 的架构师，后端设计的绝对核心原则只有一个：**坚决防止大模型的“幻觉”污染世界的客观物理状态。**
为了实现这一点，我们不能采用传统的“请求-响应”Web 架构，而必须采用\*\*“异步时钟驱动 + 批处理状态机”\*\*的架构。
以下是为 Project Petri 定制的后端架构设计方案。

***

### 一、 架构全景图

```text
[前端上帝视角 UI]
       │ ▲ WebSocket (实时日志/状态流)
       │ ▼ HTTP (上帝干预指令)
=================[ API Gateway : FastAPI ]=================
       │
       ▼
[ Tick 调度器 ] ◄─────── 全局心跳控制器
       │
       ├─(1)──► [ 环境引擎 ]: 推进时间, 饥饿度衰减, 天气变化
       │
       ├─(2)──► [ 感知广播器 ]: 扫描每个 Agent 周围, 生成视野内文本
       │
       ├─(3)──► [ AI 认知管线 ] (并行异步处理每个 Agent)
       │             │
       │             ├─► [ 记忆检索 RAG ]: 从 VectorDB 拉取相关历史
       │             ├─► [ Prompt 组装器 ]: 拼接(人设 + 记忆 + 感知 + 规则约束)
       │             ├─► [ 模型路由器 ]: 闲聊走本地, 冲突走 GPT-4o
       │             └─► [ 结构化解析器 ]: 强制提取 JSON 意图 (防幻觉第一道闸)
       │
       ├─(4)──► [ 规则引擎 ]: (核心!) 纯逻辑校验意图合法性 (防幻觉第二道闸)
       │             │
       │             └─► [ Wasm 沙箱 ]: 处理复杂合成/伤害计算 (可选)
       │
       ├─(5)──► [ 状态裁决器 ]: 修改权威的 WorldState (扣血, 移动, 转移物品)

       ▼

```

### 二、 核心模块详细设计

#### 1. 状态管理层

世界只有一个真理，它不归大模型管，归数据结构管。

- **WorldState (全局状态)**：使用纯 Python 数据类（或 Pydantic 模型）驻留在内存中。
  - `tick_count: int` (当前时间)
  - `nodes: Dict[str, LocationNode]` (地图节点，包含当前在节点内的 Agent ID 列表)
  - `items: Dict[str, Item]` (世界上所有物品的绝对归属，物品必须在某个 Node 或某个 Agent 的 inventory 里)
- **AgentState (个体状态)**：
  - `base_stats`: HP, Hunger, Money (纯数字)。
  - `personality_vector`: `[aggression: 0.8, greed: 0.5]` (影响 Prompt 的隐藏参数)。
- **持久化策略**：不做细粒度的数据库读写（太慢）。每 100 个 Tick，将整个 `WorldState` 序列化为一个 JSON 文件做**快照** 存盘。崩溃时回滚到上一个快照。

#### 2. Tick 调度器

这是世界的心跳。**采用“收集意图 -> 批处理执行”模式，绝对禁止边问大模型边改状态**（否则会产生严重的并发时序漏洞）。

- **Step 1 (环境步进)**：时间+1，所有 Agent 饥饿度 -1。
- **Step 2 (并行思考)**：使用 `asyncio.gather()`，把当前时刻所有活跃 Agent 的状态打包，**同时发给 LLM** 去思考下一步动作。
- **Step 3 (串行结算)**：等待所有 LLM 返回 JSON 意图后，在内存里**按顺序**一条条过规则引擎，更新 `WorldState`。比如 A 杀了 B，B 就变成 Dead 状态，后续轮到 B 时直接跳过。

#### 3. 规则引擎 —— 物理法则的捍卫者

这是整个项目最重要的一段代码，**这里绝对不能调用任何 AI API**。

- **设计模式**：策略模式。定义一个基类 `BaseActionValidator`，所有动作继承它。
- **执行逻辑**：
  ```python
  class AttackValidator(BaseActionValidator):
      def validate(self, agent: AgentState, intent: dict, world: WorldState) -> Judgement:
          target_id = intent.get("target_id")
          weapon_id = intent.get("weapon_id")

          # 1. 目标存在性校验
          if target_id not in world.get_nearby_agents(agent.location):
              return Judgement(success=False, reason="目标不在视野内")

          # 2. 前置条件校验 (防幻觉核心)
          if weapon_id not in agent.inventory:
              return Judgement(success=False, reason="你没有这把武器", forced_action="idle")

          # 3. 效果计算 (纯数学)
          damage = world.items[weapon_id].damage
          return Judgement(success=True, state_delta={"target_hp": -damage})
  ```

#### 4. AI 认知管线

如何把冰冷的数据变成大模型能听懂的人话，又如何把大模型的话变成冰冷的数据。

- **输入端 (感知构建)**：
  不要把整个世界扔给大模型。只拼装它视线内的东西。
  _Prompt 示例_：“你是铁匠。你很饿(饥饿度80)。你看到酒保在柜台后。你口袋里有10块钱。你昨天被酒保骗过钱(记忆检索)。”
- **输出端 (结构化解析)**：
  使用 `instructor` 库或强力的 Pydantic `model_validate`，结合正则表达式。
  如果大模型回复：“嘿，我要打他！我掏出我的火箭筒轰他！”
  解析器提取出：`{"action": "attack", "weapon": "rocket_launcher"}`。
  然后把这个 JSON 丢给上面的【规则引擎】，规则引擎发现没有火箭筒，直接打回重试或强制改为空手打。

#### 5. 记忆与 RAG 系统

长线演化的核心。

- **存入**：每个 Tick 结算后，提取三元组 `<主语, 谓语, 宾语, 情感>`。例如：“铁匠 攻击 酒保 (愤怒)”。向量化后存入 ChromaDB，打上 `agent_id` 和 `timestamp` 标签。
- **取出**：Agent 决策前，搜索条件：`where agent_id='blacksmith' and keywords=['酒保']`。取回 Top 3 相关记忆，塞进 Prompt。
- **遗忘机制**：ChromaDB 的 metadata 里记录时间，检索时对太久的记忆降权，或者直接用定时任务清理无价值的废话记忆。

### 三、 技术栈终极确认

作为极客开源项目，拒绝臃肿，追求极致的控制力：

| 模块        | 技术选型                        | 理由                                                     |
| :-------- | :-------------------------- | :----------------------------------------------------- |
| **主干框架**  | **FastAPI + Pydantic v2**   | 原生异步，Pydantic 是强类型校验和防幻觉的绝对利器。                         |
| **并发控制**  | **asyncio + asyncio.Queue** | 轻量级处理高并发的 LLM 请求，无需引入复杂消息队列。                           |
| **大模型接入** | **LiteLLM**                 | 一个库统一调用 OpenAI/Claude/本地 Ollama 的接口，极其方便做模型降级路由。       |
| **记忆存储**  | **ChromaDB** (本地持久化模式)      | 纯 Python，对极客友好，零外部依赖（不需要装 Docker 跑 Milvus），适合开源项目一键跑起。 |
| **安全沙箱**  | **Wasmtime (Python 绑定)**    | 后期如果允许 Agent 写代码造物，用来隔离执行环境。                           |
| **状态快照**  | **SQLite / JSON 文件**        | 用来存盘世界状态和日志，方便日后回放世界历史。                                |

### 四、 第一个 Sprint 的代码结构建议

不要一上来就建微服务，就在一个 Python 工程里按以下目录结构组织：

```text
petri/
  src/
  ├── core/
  │   ├── engine.py          # Tick 调度器 (世界心跳)
  │   ├── world_state.py     # 权威状态定义
  │   └── rule_engine.py     # 规则裁决器 (纯逻辑)
  ├── agents/
  │   ├── base_agent.py      # Agent 数据结构
  │   ├── cognitive.py       # 感知、Prompt 拼装
  │   └── parser.py          # LLM 输出 JSON 强制解析
  ├── memory/
  │   └── chroma_store.py    # RAG 记忆存取逻辑
  ├── llm/
  │   └── router.py          # LiteLLM 封装，负责分配模型
  ├── api/
  │   └── main.py            # FastAPI 入口，提供 WebSocket
  ├── snapshots/             # 世界存档目录
  └── tests/
      └── test_rules.py      # 极其重要！规则引擎的单元测试
```
